# Chapter 1:Reliable, Scalable, and Maintainable Applications

### Basic Definitions
* **Reliability**: able to work in spite of hardware faults, software faults, and human error or malicious use
* **Scalability**: able to handle increasing loads (data, traffic, complexity, etc)
* **Maintainability**: enables people to adapt the system to new requirements 
* **fault**: a component of system deviates from spec
* **failure**: system as a whole stops providing required service to user

### Limiting Factors of Modern Systems
* data-intensive apps

## Reliability

#### Why Reliability Matters?
* downtime or service disruptions can lead to lost productivity and revenue
* data or service loss can lead to loss of public trust or even legal ramifications

#### Reliability Tradeoffs
* Tradeoff of more dev speed e.g prototyping products for unproven markets
* Tradeoff for lower operational cost e.g for low-margin products and services, phasing out support for legacy services


### Hardware Faults
* Signs of Hardware Faults:
    * 500s errors when sending API calls
    * isolated incidents
* Typical Examples:
    * power grid black-outs
    * hard disk crash
    * RAM losing efficacy
* Typical solutions:
    * adding redundancy: backup generators and other backups
    * software fault-tolerance techniques to handle loss of entire machines
        * e.g exponential backoff to handle 500 internal server errors
        * rolling upgrades to deploy on one machine at a time in a system instead of having downtime on the entire system

### Software Errors
* Signs of Software Errors
    * Systemic errors that are happening across multiple instances or components in the system
    * tends to repeat
    * triggered by specific input
    * often receiving 400s or 500s errors when sending API calls
* Examples:
    * software bug that causes fault or failure when application receives specific input
    * a runaway process using up a shared resource: CPU, memory, disk space, network bandwidth
    * a service slows down, becomes unresponsive, or starts returning corrupted responses
    * cascading failures
* Solutions:
    * thorough testing
    * constant measuring, monitoring, and analyzing systems in production

### Human Errors
* Examples:
    * configuration errors
    * hacker activity
    * DoS due to spikes in traffic
* Solutions:
    * well-designed abstractions, APIs, admin interfaces that enable users to use application for intended uses
    * decouple areas where people make mistakes from where they can cause failures
        * e.g sandbox environment for devs testing out new implementations, or testing
    * Test thoroughly at all levels: unit tests, whole system integration tests, automated tests, manual tests
    * Allow quick, easy recovery from human errors
        * be ready to roll back
        * do rolling updates
        * be able to recompute data 
            * e.g ability replay messages from data stream, database, cache
    * detailed and clear monitoring, i.e telemetry
        * have dashboard with performance metrics, error rates
        * maybe have alarms for unexpected behavior, especially if it doesn't necessarily trigger a failure
    * good management practices and training


## Scalability

### Load Parameters: Define the relevant conditions for scalability
* Considerations:
    * What load does your system need to handle? (Very often, it's a certain number of requests per unit of time)
        * web server: requests/second
        * database: read-to-write ratio
        * cache: hit rates (number of reads per second)
    * What edge cases can cause a bottleneck, and what do you measure to define those edge cases?
        * e.g distribution of followers per user, in Twitter's fan-out architecture
            * celebrities with high follower counts are edge cases because their tweets require substantially more work to deliver to all their followers' timelines

### Performance metrics: measure the impact of scaling
* Considerations:
    * When you increase a load parameter while keeping all your system resources (CPU, memory, bandwidth, etc) unchanged, what gets impacted?
        * typically **response time**:time between client sending a request and receiving a response
        * for batch processing systems, consider **thoroughput**: number of records processed per second
    * When you increase a load parameter, what system resources do you augment to keep performance the same?

### Interpeting Statistics
* percentiles give a fuller picture than averages
    * medians: 50% of user experiences fall below and above it, a good estimate of the typical user experience 
    * high percentiles (95th, 99th, 99.9th): describe outliers
        * can sometimes be the most critical cases, e.g Amazon's 99.9th percentile response times come from people who have the most data (i.e purchases) on their accounts and are Amazon's most valuable customers
* response times on client vs server side can have major discrepancies
    * e.g **queuing delays** due to **head-of-line blocking**: queuing delays in the first few records in the queue increases the response times of subsequent records
    * **tail latency amplification**: when you make parallel back-end calls for an end-user request, you need to wait for the slowest call to complete and those slow calls get more likely the more backend calls each end-user request requires
    * you should test with asynchronous loading: sending records without waiting for response to catch issues like queuing delays and tail latency amplification

### Tactics to Handle More Load
* Vertical scaling vs horizontal scaling
    * vertical scaling: using more powerful machines, can be more expensive
        * typical approach until costs exceed benefit
    * horizontal scaling: using more machines
        * pro: simple for stateless systems
        * con: can be complicated for stateful systems
    * typically, enterprises take a hybrid approach
* Elastic vs Manual systems
    * elastic systems automatically add computing resources if load increase detected
        * e.g AWS load balancers
        * pros: useful for highly unpredictable load
    * manual systems allow human to analyze capacity and add more computing resources if needed
        * pro: simpler, less operational surprises
* Planning and Design
    * you want to determine your assumptions about what operations will be common and what will be rare i.e the load parameters

## Maintainability

### Includes
* fixing bugs
* keeping systems operational
* investigating failures: logging
* adapting to new platforms
* modifying to new use cases, adding features
* repaying technical debts

### Design Principle #1: Operability
* make it easy for operations teams to keep system running smoothly
* operations teams' responsibilities:
    * monitoring health of system and quickly restoring an unhealthy system
    * tracking down cause of problems e.g system failures, degraded performance
    * keeping software and platforms up to dates e.g security patches

| knowing how different systems affect each other to avoid problematic changes |
| anticipate and solving future problems i.e capacity planning | |
| establish good practices for deployment, configuration, management, etc| |
| complex maintenence tasks like migrations | 
| maintain security as configs change | 
| keep production environment stable |
| 

### Design Principle #2: Simplicity
* make it easy for new engineers to understand the system, particularly by minimizing complexity

### Design Principle #3: Evolvability
* make it easy for engineers to make changes to the system in the future aka extensibility, modifiabilty, plasticity
